Basics of Node.js

What is Node?

  • Javascript runtime environment
  • Built on Chrome's V8 engine
  • Event-driven
  • Non-blocking I/O
  • Single-threaded
Switch to terminal and run basic JS in REPL

Synchronous execution


function getData(id) {
  var data = readFile('/path/to/dir/' + id);
  return data;
}

var myData = getData(123);
console.log(myData);
  • Fine in a multi threaded system
  • Works in Node - but blocked while reading each file
  • Can't serve more requests until the current one is done
Node is single threaded, so any long running work which is blocking will cause a lack of throughput in an app

Asynchronous execution


function getData(id, callback) {
  var data = readFile('/path/to/dir/' + id);
  callback(data);
}

var myData = getData(123, function(data) {
  console.log(myData);  
});
  • Node can execute this asynchronously
  • Callback is called when ready, then Node processes it
Node is single threaded, so any long running work which is blocking will cause a lack of throughput in an app

Middleware Pattern

  • Design pattern used by Express
  • Similar to Filter Chain Pattern in Java

What is Middleware?

  • Functions which have access to the request, response, and next, piece of middleware in the chain

function(req, res, next) {
  // ...
};

What is Middleware?

  • Middleware can:
    • Execute code
    • Alter the request and response objects
    • End the request
    • Call the next middleware function in the chain
  • If a piece of Middleware does not either end the request, or call next(), the request will be left hanging

Types of Middleware

  • Application-level Middleware

app.use(function(req, res, next) {
  console.log('The time is: ' + Date.now());
  next();
});
  • This will be called on all requests into the app

Types of Middleware

  • Router-level Middleware

app.get('/hello', function(req, res, next) {
  console.log('Hello to you too');
  next();
}, function(req, res){
  res.send('Hello, World!');
});
  • Bound to requests which match this route
Note the use of two bits of middleware here

Types of Middleware

  • Error-Handling Middleware

app.use(function(err, req, res, next) {
  console.log(err);
  res.status(500).send('Error!!!');
});
  • Separate from the main middleware chain
  • Four arguments makes Express use this as error middleware
  • Invoked if you call next with a parameter - e.g. next('error')
If you invoke error middleware, all other middlewares are skipped over Note the need for the next parameter, even if it isn't used, for Express to interpret this as error middleware.

Girdle

What is Girdle?

  • A service which returns a Handlebars template
  • Contains all site scaffolding
    • Basic layout of the site
    • <head> and <body> mark up
    • Header, Nav, and Footer mark up
    • All global styles and scripts
    • Hooks for pages to populate

Demo
Demo Girdle - explain Hooks

Important Hooks

  • {{content}} - Where the page content should go
  • {{meta.title}} - Page title
  • {{meta.description}} - Page description
  • {{head.css.after}} - Links to page specific CSS files
  • {{footer.scripts.after}} - Links to page specific JS files

Variations

  • Base URL: /girdle/fabric
  • Different template per country/language
    • /girdle/fabric?locale=zh-CN
  • "Surface Analytics"
    • /girdle/fabric?surfaceAnalytics=true
    • If true, won't load old analytics library
    • Loads jQuery at the bottom of the page
  • "Canonical URL"
    • /girdle/fabric?canonicalUrl=true
    • Provides meta canonical URL tag
    • Compulsory for new apps - Soon to be removed

Cache Headers

  • Girdle provides cache headers, which should be honoured
    • Cache-Control: max-age=3600
    • Cache-Control: stale-if-error=259200
    • Cache-Control: stale-while-revalidate=120

AWS

S3

  • Cloud storage
  • Ideal for static, or rarely changing files
  • Use to store static assets for the website
  • NOT a fast delivery mechanism
  • Use a CDN in front for fast service/caching

EC2

  • Virtual Machines when you need them
  • Launch more when you need them, tear them down when you don't
  • Pre-configured Amazon Machine Images (AMIs) - OS, software, config etc
  • Different combinations of CPU, memory, storage, network capacity
  • Easy to scale

Elastic Beanstalk

  • Application Container
  • Deploy and run your application easily
  • Out the box Elastic Load Balancers
  • Out the box Auto-scaling EC2 instances
  • Full monitoring of your environment built-in
  • Easy configuration
  • Integration with CloudWatch for logs/metrics/alarms

CloudWatch

  • Real time monitoring of AWS resources
  • Monitor CPU, latency, request counts
  • Add custom monitoring using CouldWatch Logs
  • Set up Alarms based on any metrics
  • Easy-to-setup dashboards
